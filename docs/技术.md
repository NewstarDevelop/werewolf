# 狼人杀 AI 单机版技术实现文档

## 1. 技术栈选型建议

鉴于本项目具有“单客户端”、“高并发AI调用”、“重后端状态管理”的特点，推荐以下技术栈：

- **前端 (Client)**: Vue 3 或 React (纯文本交互，UI逻辑较轻，重点是聊天流展示)。
- **后端 (Server)**: **Python (FastAPI + WebSockets)**。推荐 Python 的原因是其对大模型生态（如 OpenAI SDK, LangChain）支持极佳，且异步协程（asyncio）非常适合处理多AI并行请求和状态机等待。
- **大语言模型 (LLM)**: 推荐 **DeepSeek-V3** (性价比极高，逻辑推理极强) 或 **Claude 3.5 Sonnet / GPT-4o**。
- **通信协议**: **WebSocket**。游戏流转需要服务器主动向客户端推送消息（如“AI正在思考”、“天亮了”），HTTP轮询不够优雅，WebSocket是最佳选择。

## 3. 核心数据结构设计 (类定义参考)

在后端代码中，建议采用面向对象（OOP）的方式建立核心数据结构。

### 3.1 玩家基类与子类

codePython

```
class Player:
    seat_id: int
    role: str           # "VILLAGER", "WOLF", "SEER", "WITCH", "HUNTER"
```

## 4. 通信协议设计 (WebSocket Payload)

前后端所有的交互通过 JSON 格式的消息进行传递，必须包含 type 和 data。

### 4.1 Server -> Client (后端推给前端)

- **SYSTEM_MSG**: 系统广播（如：“天黑请闭眼”）。
- **CHAT_UPDATE**: 发言更新（如：“3号玩家发言：...”）。
- **AI_THINKING**: AI思考状态指示（前端显示：🤖 3号正在输入...）。
- **REQUIRE_INPUT**: 轮到真人操作，解锁UI。
  - *Payload示例*: {"type": "REQUIRE_INPUT", "data": {"action_type": "SPEAK"}}
- **GAME_OVER**: 游戏结束结算。

### 4.2 Client -> Server (前端发给后端)

- **SUBMIT_ACTION**: 真人提交操作（发言文本、杀人目标、投票目标）。
  - *Payload示例*: {"type": "SUBMIT_ACTION", "data": {"action": "VOTE", "target": 5}}

## 5. LLM Agent 模块设计 (核心难点)

如何让AI不乱说话、严格遵守游戏规则？必须通过 **System Prompt (系统指令)** 和 **结构化输出约束** 来实现。

### 5.1 Prompt 结构组装

每次向LLM发起请求时，发送的 Prompt 应包含三部分：

1. **系统规则 (System Prompt)**:
   - *“你是一个正在玩9人局狼人杀的AI玩家，你的座位号是X，你的身份是Y。规则是...”*
2. **当前记忆与上下文 (Context)**:
   - *“第一天白天，存活玩家有[1,2,4,5,6,8,9]... 历史发言如下：1号说... 2号说...”*
3. **当前任务指令 (Task)**:
   - *“现在轮到你发言了。请结合场上局势进行分析推理。注意伪装你的身份。”*

### 5.2 强制 JSON 输出 (Structured Output)

**绝对不要让大模型直接输出纯文本！** 必须让它输出 JSON，方便后端截取发言内容和解析执行动作。

**AI 发言阶段的约束格式：**

codeJSON

```
{
  "inner_thought": "我是狼人，刚才4号真预言家查杀了我的狼同伴5号，我得跳出来对跳预言家保下5号，顺便把水搅浑...",
  "speech_text": "4号完全是在胡说八道！我才是真正的预言家，我昨晚查了5号，他是纯金水。4号必然是悍跳狼！"
}
```

*后端拿到返回值后，将 inner_thought 丢弃（或记录在后台日志中），只把 speech_text 通过 WebSocket 广播出去。*

**AI 投票阶段的约束格式：**

codeJSON

```
{
  "inner_thought": "4号是对跳预言家，必须把他投出去。",
  "vote_target": 4 
}
```

## 6. 核心运行逻辑 (伪代码演示)

使用 Python 的 asyncio 来处理阻塞等待。状态机的每一个环节就是一个 async 函数。

codePython

```
async def phase_day_speaking(context: GameContext):
    # 按照发言顺序遍历存活玩家
    speaking_order = get_speaking_order(context) 
    
    for seat_id in speaking_order:
        player = context.players[seat_id]
        
        if isinstance(player, HumanPlayer):
            # 通知前端解锁输入框
            await ws.send_json({"type": "REQUIRE_INPUT", "data": {"type": "SPEAK"}})
            # 阻塞等待前端返回消息事件
            human_text = await wait_for_human_submit() 
            context.public_chat_history.append(f"{seat_id}号发言: {human_text}")
            await broadcast_chat(f"{seat_id}号发言: {human_text}")
            
        elif isinstance(player, AIPlayer):
            # 广播AI思考状态
            await broadcast_status(f"🤖 {seat_id}号玩家正在思考...")
            # 组装Prompt，异步调用大模型
            prompt = build_speak_prompt(player, context)
            response_json = await call_llm_api(prompt)
            
            speech = response_json.get("speech_text", "我没什么好说的，过。")
            context.public_chat_history.append(f"{seat_id}号发言: {speech}")
            await broadcast_chat(f"{seat_id}号发言: {speech}")

    # 所有人都发言完毕，流转到下一个状态
    await transition_to("VOTING")
```

## 7. 异常处理与防崩溃机制 (容错设计)

大模型是不稳定的，必然会出现以下问题，必须在代码中写好兜底（Fallback）逻辑：

1. **AI 输出的 JSON 格式错误 / 没闭合**
   - *方案*：使用 json.loads 解析时加 try-except。如果解析失败，使用正则表达式尝试提取 {} 内的内容。如果彻底失败，触发重试机制（Retry），最多重试2次。2次仍失败，系统代为输出默认操作（如发言为“过”，投票为弃票 0）。
2. **AI 违规操作（幻觉）**
   - *场景*：AI决定杀 3号，但 3号 昨天已经死了。
   - *方案*：后端在接收到 vote_target 时必须进行合法性校验。如果目标非法（如已死亡、超出1-9范围），则触发重试，并在 Prompt 中严厉追加一条：“注意！你刚才选择的目标是非法的，只能在 [合法列表] 中选择！”
3. **API 请求超时**
   - *方案*：设置 timeout（例如 20 秒）。超时后算作该 AI 掉线一回合，默认执行“过/弃权”。
