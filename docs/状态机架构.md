# 狼人杀核心状态机架构文档 (9人局单机文字版)

## 1. 游戏基础配置

- **玩家规模**：9人（1个真人玩家 + 8个AI玩家）。
- **身份板子**：3 狼人、3 平民、3 神职（预言家、女巫、猎人）。
- **交互方式**：纯文本展示与输入。
- **发言规则**：白天死者宣布后，由系统指定起始位置，顺时针/按座位号依次发言。

------



## 2. 状态机总览 (State Machine Overview)

游戏由一个中央状态机（Game Engine）驱动，状态机的每一次流转（Transition）都会触发相应的事件、更新全局上下文记录（Context），并决定下一步是**等待真人输入**还是**请求AI大模型**。

## 3. 核心状态定义与行为解析

### 3.1 阶段一：初始化与判定

- **INIT (初始化)**
  - **行为**：洗牌，随机分配1~9号座位。真人玩家随机获得一个座位和身份，其余分配给AI。初始化系统聊天记录（Chat History）和存活状态表。
  - **流转**：自动进入 CHECK_WIN。
- **CHECK_WIN (胜利判定)**
  - **行为**：检查存活人数。
    - 若狼人全部死亡 -> 好人阵营胜利。
    - 若平民全部死亡 或 神职全部死亡 -> 狼人阵营胜利（屠边局）。
  - **流转**：若有阵营胜利进入 GAME_OVER，否则进入下一阶段（首轮进 NIGHT_START，白天结束进 NIGHT_START）。

### 3.2 阶段二：夜晚结算 (暗拍阶段)

*夜晚的核心原则是不公开任何信息，所有行动偷偷写入后端变量中。*

- **NIGHT_START (入夜)**
  - **行为**：系统输出文本：“天黑请闭眼”。重置昨夜死亡名单 killed_tonight = []。
- **WOLF_ACTION (狼人行动)**
  - **行为**：存活的狼人选择目标。
    - **如果真人是狼人**：UI 解锁，等待真人输入要杀的座号。
    - **AI狼人**：并行调用AI大模型，传入[私有视野]，请求杀人目标。
    - *逻辑处理*：综合狼人意见（由于是纯文字版简易处理，可设定为“多狼投票，票多者死”或“按某位AI狼的决定为主”）。记录目标。
- **SEER_ACTION (预言家行动)**
  - **行为**：预言家选择查验目标。
    - **如果真人是预言家**：等待真人输入目标。
    - **AI预言家**：调用大模型获取查验目标。
    - *逻辑处理*：将查验结果（好人/狼人）仅返回给预言家玩家（加入其私有上下文）。
- **WITCH_ACTION (女巫行动)**
  - **行为**：告知女巫昨夜死者（仅限女巫有解药时），选择是否用解药或毒药。
    - **如果真人是女巫**：UI提示“昨夜X号死了，是否使用解药？是否使用毒药？”等待输入。
    - **AI女巫**：调用大模型，传入死者信息，获取决策。
    - *逻辑处理*：更新 killed_tonight 列表（被救则移除，被毒则增加）。
- **NIGHT_END (夜晚结算)**
  - **行为**：合并死亡名单，标记死亡玩家状态。如果有猎人死亡，记录猎人开枪状态。

### 3.3 阶段三：白天结算 (明拍阶段)

- **DAY_START (天亮)**
  - **行为**：系统输出文本：“天亮了。昨夜死亡的是 X 号。”（或“昨夜平安夜”）。
- **DEAD_LAST_WORDS (死者遗言)**
  - **条件**：仅在第一天夜晚死亡的玩家有遗言。
  - **行为**：触发死者发言。真人则等待输入，AI则请求LLM生成遗言。
- **DAY_SPEAKING (依次发言 - ★ 最复杂状态)**
  - **行为**：系统维护一个 current_speaker（当前发言者）指针。从死者左手边（或随机指定1人）开始顺时针遍历存活玩家。
  - **内部循环逻辑**：
    1. 判断 current_speaker 是真人还是AI。
    2. **若是真人**：UI激活输入框，等待玩家提交发言文本。
    3. **若是AI**：系统将当前的 [全局聊天记录] + [该AI的私有身份指令] 组装为 Prompt，调用 LLM API，获取生成的发言文本。
    4. 将发言文本广播给所有人（追加到全局聊天记录）。
    5. current_speaker 指向下一位存活玩家。
  - **流转**：当所有存活玩家发言完毕，进入 VOTING。
- **VOTING (投票放逐)**
  - **行为**：所有存活玩家投票。
    - **真人**：UI展示投票面板。
    - **AI**：并发请求所有存活AI的投票决策（要求大模型输出要投的座位号）。
  - **流转**：收集齐9人（存活者）选票后，进入 VOTE_RESULT。
- **VOTE_RESULT (宣布放逐)**
  - **行为**：统计票数。票多者被放逐（死亡）。如果是平票，简易版规则可直接判定为“无人出局”或“随机出局一人”以简化流程。
  - **流转**：进入 BANISH_LAST_WORDS。
- **BANISH_LAST_WORDS (放逐遗言)**
  - **行为**：被放逐者发表遗言（逻辑同首夜遗言）。如果是猎人且未被毒杀，此时可触发开枪技能。
  - **流转**：进入 CHECK_WIN（天黑前判定一次胜负）。

## 4. 关键数据结构设计 (Context Data)

状态机在流转时，必须维护一个全局的 GameContext，用于生成给AI的 Prompt。

```
{
  "day_count": 1, 
  "current_phase": "DAY_SPEAKING",
  "players": [
    {
      "seat": 1,
      "role": "WITCH",
      "is_human": true,
      "is_alive": true,
      "has_potion": true,
      "has_poison": true
    },
    // ... 座位2-9
  ],
  "public_history": [
    "[系统] 游戏开始，分配身份完毕。",
    "[系统] 天黑请闭眼...",

    "[发言] 4号: 我是预言家，昨晚查验了5号，他是好人..."

    ]
  }
}
```

## 5. 针对 1真人 + 8AI 的特殊架构建议

在代码实现时，最大的挑战是 **“同步的游戏流程” 与 “异步的AI网络请求” 之间的协调**。

1. **阻塞与唤醒机制 (Async/Await)**：
   - 状态机应该运行在一个异步函数中。
   - 当轮到真人行动时，状态机 await 一个前端发送的 SubmitEvent。
   - 当轮到AI行动时，状态机 await LLM API 的返回结果。
2. **AI 的思考时间掩盖 (UX优化)**：
   - 在纯文字版中，轮到AI发言时，前端不要干等着。
   - 系统应立刻输出："🤖 2号玩家正在思考中..."。
   - 当收到LLM返回的内容后，再替换为："[发言] 2号: 我觉得刚才4号说的不对..."。
3. **结构化输出 (强制AI遵守规则)**：
   - 不要让AI直接返回纯文本，这极容易导致游戏崩溃（例如它在投票阶段回复“我觉得5号是狼”而不是数字“5”）。
   - 强烈建议在 Prompt 中要求大模型返回 **JSON 格式**。
   - *示例 Prompt 约束*：“请先在 thought 字段输出你的推理过程，然后在 speech 字段输出你对外说的话。格式必须为 {"thought": "...", "speech": "..."}”。前端解析JSON后，只将 speech 展示给真人玩家。
