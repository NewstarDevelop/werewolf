  日志监控问题汇总

  问题 1：WebSocket 连接反复断开重连（严重）

  后端日志显示整个测试期间共发生 11 次 WebSocket 连接，其中 10 次都是 open 后很快
  close。每次断开后前端自动重连（scheduleReconnect，间隔 1 秒），形成循环。

  根因分析：后端 game_socket 在连接建立后立即 create_task(run_game_session(...))，游戏引擎跑完（AI 全部用的是本地默认
  provider，几乎瞬间完成）后，如果没有人类输入阻塞，引擎很快结束。引擎结束后 WebSocket 的 while True 循环仍在等
  receive_json，但游戏已经结束了——没有机制通知前端游戏已完结并优雅关闭连接。同时前端在 close
  事件里无条件触发重连，导致循环。

  问题 2：前端重连不重置游戏状态

  App.tsx:226 — useEffect 依赖 connectionAttempt，每次重连会创建新 WebSocket，但 entries 和 players
  状态没有重置。这意味着重连后旧的聊天记录和玩家状态会累积叠加，UI 会越来越混乱。

  问题 3：游戏结束后缺少连接关闭逻辑

  routes.py:356-358 — run_game_session 发送 GAME_OVER 后直接 return，但没有关闭 WebSocket。后端继续在 while True
  里等消息，前端收到 GAME_OVER 后也没有主动断开。游戏结束后连接处于"僵尸"状态。

  问题 4：前端 UI 仍显示骨架占位文案

  App.tsx:248-249 —
  页面标题写的是"工程骨架已启动"和"当前阶段先完成前后端骨架..."，这是开发阶段的占位文案，应该替换为正式的游戏界面。

  问题 5：无错误/异常日志输出

  整个测试期间后端没有任何 ERROR 或 WARNING 级别日志，只有
  INFO。这说明没有配置结构化日志，游戏引擎内部的异常（如果有）会被静默吞掉，不利于排查问题。
