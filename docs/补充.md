### 一、 核心逻辑查漏补缺 (Review & Refine)

#### 1. 猎人开枪技能的“状态机缺失”

在你的状态机设计中，提到“如果有猎人死亡，记录猎人开枪状态”，并在遗言阶段“可触发开枪技能”。
**问题**：猎人开枪是一个**会改变场上存活人数**的“强干预”动作，且开枪后被带走的人如果是特定身份（比如狼人或平民），可能会瞬间触发 CHECK_WIN。
**修正方案**：在状态机中增加一个独立的 HUNTER_SHOOTING 状态。

- 流程：宣布死讯 -> 检查死者中是否有带枪猎人 -> 若有，进入 HUNTER_SHOOTING -> 等待猎人（真人或AI）选择目标 -> 结算开枪死亡 -> CHECK_WIN -> 返回正常的 DEAD_LAST_WORDS 或 DAY_SPEAKING。

#### 2. “多狼在夜间如何统一意见”的落地细节

文档中提到“综合狼人意见”。但在并发调用 3 个 AI 狼人时，它们极有可能选出 3 个不同的杀人目标，这会让游戏逻辑卡死。
**修正方案（简易且高效）**：

- **若真人是狼**：以真人玩家的选择为绝对优先级（“你是狼王”），另外两个 AI 狼只做形式上的附和。
- **若无真人狼**：后台指定一名存活的 AI 狼为“首领狼（Alpha Wolf）”，只请求这一个 AI 的刀人目标，其他两狼继承该目标。（简化并发复杂度，且符合游戏结果）。

#### 3. LLM 的 JSON 解析崩溃兜底

虽然文档提到了用 Regex 尝试提取 JSON，但基于目前的大模型生态，有更优雅的做法。
**优化方案**：

- 如果使用 **DeepSeek-V3 / OpenAI**，在 API 请求时强制加上 response_format={ "type": "json_object" }。这能从引擎底层保证输出绝对是合法的 JSON，免去手写 Regex 提取的麻烦。
- 结合 Python 的 Pydantic 库来做严格的数据校验。

#### 4. “视野隔离”（Viewpoint Masking）的致命危险

在生成 Prompt 时，千万注意**不能把全局 GameContext 完整扔给 AI**。AI 非常聪明，如果在 Prompt 的犄角旮旯不小心带上了“4号是女巫”的信息，它会立刻利用这一点。
**优化方案**：必须在后端写一个 get_player_view(seat_id, context) 函数，专门对全局 Context 进行**脱敏过滤**。

### 二、 核心难点实现方案：如何让“状态机”停下来等“真人”？

在后端的 FastAPI 异步架构中，最难处理的是：**状态机正在跑一个 while True 循环，当轮到人类时，它怎么暂停并等待 WebSocket 传回来的消息？**

建议使用 Python 的 asyncio.Future。以下是为你量身定制的核心代码骨架：

#### 1. 玩家类定义 (携带 Future 锁)

codePython

```
import asyncio
from fastapi import WebSocket

class Player:
    def __init__(self, seat_id: int, role: str):
        self.seat_id = seat_id
        self.role = role
        self.is_alive = True

class HumanPlayer(Player):
    def __init__(self, seat_id: int, role: str, ws: WebSocket):
        super().__init__(seat_id, role)
        self.ws = ws
        # 核心：用于阻塞状态机等待人类输入的“锁”
        self.pending_input: asyncio.Future = None 

class AIPlayer(Player):
    def __init__(self, seat_id: int, role: str, personality: str):
        super().__init__(seat_id, role)
        self.personality = personality
```

#### 2. WebSocket 路由层 (负责解锁 Future)

codePython

```
@app.websocket("/ws/game")
async def websocket_endpoint(ws: WebSocket):
    await ws.accept()
    
    # 假设人类玩家分配在 1 号位
    human = HumanPlayer(seat_id=1, role="SEER", ws=ws)
    game_engine = GameEngine(human)
    
    # 启动状态机（在后台独立运行）
    asyncio.create_task(game_engine.run_loop())

    try:
        while True:
            # 持续监听前端发来的消息
            data = await ws.receive_json()
            
            if data["type"] == "SUBMIT_ACTION":
                # 当收到人类操作时，如果状态机正在等待人类，则把数据传给 Future 解锁！
                if human.pending_input and not human.pending_input.done():
                    human.pending_input.set_result(data["data"])
                    
    except WebSocketDisconnect:
        # 处理断线逻辑
        pass
```

#### 3. 状态机流转层 (阻塞等待逻辑)

codePython

```
class GameEngine:
    async def run_loop(self):
        self.state = "DAY_SPEAKING"
        while self.state != "GAME_OVER":
            if self.state == "DAY_SPEAKING":
                await self.handle_day_speaking()
            # 其他状态...

    async def handle_day_speaking(self):
        for seat_id in self.speaking_order:
            player = self.players[seat_id]
            if not player.is_alive: continue

            if isinstance(player, HumanPlayer):
                # 1. 告诉前端：该你了，解锁输入框
                await player.ws.send_json({"type": "REQUIRE_INPUT", "data": {"action": "SPEAK"}})
                
                # 2. 创建一个 Future 锁
                player.pending_input = asyncio.get_running_loop().create_future()
                
                # 3. 停在这里死等！直到 WebSocket 路由层收到消息并 set_result
                try:
                    # 加入超时机制，防止人类挂机
                    human_action = await asyncio.wait_for(player.pending_input, timeout=60.0)
                    speech_text = human_action.get("text", "过。")
                except asyncio.TimeoutError:
                    speech_text = "我没什么想说的，过。" # 挂机兜底
                    
                # 4. 广播人类发言
                await self.broadcast(f"{seat_id}号发言：{speech_text}")

            elif isinstance(player, AIPlayer):
                # 广播AI思考中
                await self.broadcast(f"🤖 {seat_id}号正在思考...")
                # 异步调用大模型（不会阻塞整体服务，只会等待当前AI发言）
                speech_text = await call_llm(player, self.context)
                await self.broadcast(f"{seat_id}号发言：{speech_text}")
                
        self.state = "VOTING" # 所有人发言完毕，进入投票阶段
```

为了达到你文档中要求的“沉浸感”和“不打破第四面墙”，你的 System Prompt 必须非常精细。推荐采用以下结构：

```

【场上局势】
当前存活玩家：{alive_list}
昨夜死亡：{dead_list}

【历史发言】

【你的私有记忆】
{private_memory}

- 如果你是好人：找出你认为的狼人，并呼吁大家投票给他。
- 如果你是狼人：伪装成好人（你可以悍跳神职），把水搅浑，或者顺势踩那些被怀疑的好人。

}
```
