# 产品需求文档 (PRD)：AI 狼人杀单机文字版 (9人局)

## 1. 产品概述

### 1.1 产品愿景

开发一款纯文本交互的单机狼人杀游戏。玩家无需等待真人匹配，可随时随地开启一局由1名真人与8名由大语言模型（LLM）驱动的AI玩家共同参与的高质量逻辑推理对局。

### 1.2 核心体验目标

- **无需等待**：秒开局，随时退出。
- **沉浸感**：AI的发言需符合人类逻辑，具有伪装、倒钩、悍跳等战术行为。
- **极简交互**：通过类似“微信群聊”的纯文本界面完成核心游戏流程。

------



## 2. 游戏规则设定（产品基线）

为了降低开发初期的复杂度，本产品严格采用以下特定规则（后续版本迭代再考虑自定义）：

- **对局人数**：9人（固定 1名真人 + 8名AI）。
- **身份板子**：3 狼人、3 平民、3 神职（预言家、女巫、猎人）。
- **胜负判定**：屠边局。即狼人阵营全灭则好人胜；平民全灭 或 神职全灭则狼人胜。
- **发言规则**：白天按座位号（死者左手边）顺时针依次发言，无抢麦机制。
- **女巫规则**：女巫全程不可自救；每晚只能使用一瓶药。
- **同票规则**：若白天投票出现平票，不进行PK发言，直接判定该轮**无人出局**（简化流程）。

------



## 3. 核心界面需求 (UI/UX)

产品整体采取“单屏操作面板”布局，无需复杂的页面跳转。主要分为三个区域：

### 3.1 左侧 / 顶部：玩家状态区 (Player List)

- 展示 1~9 号玩家的座位图标。
- 每个座位需显示：
  - 存活状态（亮起/置灰并加上“墓碑”标识）。
  - 玩家标识（标明哪个是“你”，并展示你的身份图标，其他AI为问号）。
  - 警长标识（暂不开发警长功能，可预留）。
  - **思考状态动画**：当轮到某位AI操作时，其头像旁显示类似 [...] 或 思考中... 的动态提示。

### 3.2 中央：全局对局日志区 (Chat History)

- 类似聊天窗口，从上往下滚动。
- 用不同颜色区分消息类型：
  - **系统播报（灰色）**：“天黑请闭眼”、“昨夜3号玩家死亡”。
  - **私人信息（蓝色）**：“你的查验结果是：5号是狼人”。
  - **玩家发言（黑色/白色气泡）**：“2号玩家：我是预言家，昨晚查杀了8号...”。

### 3.3 底部：用户操作区 (Action Panel)

- 根据当前阶段，动态切换面板状态：
  - **等待状态**：展示“等待其他玩家行动...”，禁用所有输入。
  - **技能/投票状态**：展示 1~9 号的数字按钮（仅展示存活且合法的目标），外加一个“确认”按钮。女巫阶段需展示“用解药/用毒药/不操作”按钮。
  - **发言状态**：展示文本输入框、字数统计（如 0/200），以及“发送发言”按钮。

------



## 4. 核心功能描述清单 (Features)

### 功能 1：游戏大厅与初始化

- **F1.1 创建游戏**：用户点击“开始游戏”，系统瞬间生成一局游戏。
- **F1.2 身份分配**：系统随机将9个身份分配给9个座位。向前端下发真人玩家的座位号和身份牌。
- **F1.3 人设生成 (可选/进阶)**：系统在后台为8个AI分配不同的初始性格（如：暴躁老哥、逻辑大师、划水怪），以增加发言多样性。

### 功能 2：暗拍阶段（夜晚流程）

- **F2.1 狼人刀人**：
  - 若真人是狼，弹出按钮选择目标。
  - 若全是AI狼，系统在后台综合所有狼人的目标，最终锁定一个受害者。
- **F2.2 预言家验人**：只能选择非自己的存活玩家，系统返回“好人”或“狼人”。
- **F2.3 女巫用药**：
  - 若出现死者且女巫有解药，系统提示死者编号，询问是否救。
  - 若无解药或已救人，询问是否对某位玩家使用毒药。

### 功能 3：明拍阶段（白天流程）

- **F3.1 死亡结算与遗言**：首夜死者可发表遗言。系统按需触发发言逻辑。
- **F3.2 依次发言**：
  - 系统严格按顺序轮流赋予玩家“发言权”。
  - AI 发言时，前端必须有1~3秒的缓冲等待感，并在获取大模型回复后上屏。
  - 真人发言时，支持打字输入，若超时（建议设定60秒）未输入，系统自动代发“过”。
- **F3.3 投票放逐**：
  - 所有存活玩家选定1名存活玩家，或选择“弃票”。
  - 系统统计票数，将得票最多者标记为死亡（放逐），并触发被放逐者的遗言环节。

### 功能 4：胜负判定系统

- **F4.1 实时校验**：任何导致玩家死亡的动作（狼刀、毒药、放逐）生效后，立即触发胜负校验。
- **F4.2 游戏结算面板**：一旦触发胜利条件，中断后续流程，弹出结算框。
  - 展示胜利阵营。
  - 揭晓 1~9 号所有玩家的真实身份。

------



## 5. 针对 AI 大模型的性能与内容需求

作为本产品的核心竞争力，对接的 LLM 必须满足以下需求：

- **响应时间限制**：单个AI生成发言文本的 API 响应时间需控制在 **5秒以内**（否则严重影响游戏体验，若大模型太慢，需使用流式输出 Stream 逐字显示）。
- **发言长度限制**：必须在 Prompt 中限制AI发言在 **50~150字** 之间，禁止长篇大论。
- **人设一致性**：AI 在整局游戏中不得忘记自己的身份（如前一轮跳平民，后一轮自己承认是狼人）。
- **禁止打破第四面墙**：AI 发言必须完全沉浸在游戏世界中，绝对不能出现诸如：“作为一个AI语言模型，我无法判断谁是狼人”、“我的JSON输出格式如下”等破坏体验的话。

## 6. 异常与边界情况处理机制 (Edge Cases)













| 场景描述                                | 期望系统行为（容错处理）                                     |
| --------------------------------------- | ------------------------------------------------------------ |
| **人类玩家强制刷新页面/关闭浏览器**     | **MVP阶段**：不保存进度，直接视为游戏作废，下次进入重新开局。<br>**未来进阶**：使用 LocalStorage 保存 GameContext，支持断点重连。 |
| **大模型 API 调用超时或报错**           | 系统重试一次，若依然失败，该AI强制执行默认操作（发言：“我没什么线索，过。” / 投票：弃票）。保证游戏主流程不卡死。 |
| **大模型尝试作弊** (如输出要杀已死的人) | 后端过滤校验。发现不合法指令，判定为废票或自动改选随机合法目标。 |
| **猎人被女巫毒死**                      | 系统后台必须能区分“被毒死”和“被刀/投死”，若被毒死，系统直接剥夺其开枪权利，不触发开枪阶段。 |

------



## 7. 验收标准 (Acceptance Criteria)

开发完成后，通过以下标准即视为 MVP（最小可行性产品）开发成功：

1. **流程完整性**：能够不产生 Bug 地跑完一局从“分配身份”到“弹出游戏结束”的完整流程。
2. **AI 基本智商**：
   - AI 预言家查出狼人后，白天知道发起攻击号召大家投票给该狼人。
   - AI 狼人不会在白天主动暴露自己的狼同伴。
3. **UI 流畅度**：真人玩家明确知道当前自己该看还是该操作，界面不出现卡死或错位现象。
